---
title: Tool Gating
description: Role-based access, prerequisite workflows, and rate limits
---

Gates control who can call a tool and under what conditions. When a gate blocks, the agent gets a structured error with enough context to self-correct.

## Gate Types

### Role-Based

Check claims from the auth token:

```ts
import { role } from "@stratus/mcp-aws";

server.tool("admin_action", { gate: role("admin") }, handler);
server.tool("write", { gate: role("admin", "editor") }, handler); // any of these roles
```

### Prerequisite (Workflow Enforcement)

Require another tool to be called first:

```ts
import { requires } from "@stratus/mcp-aws";

server
  .tool("review_trade", handler)
  .tool("execute_trade", {
    tier: "hidden",
    gate: requires("review_trade"),
  }, handler);
```

When `review_trade` completes:
1. `execute_trade` is automatically promoted to visible
2. Server sends `tools/list_changed`
3. Agent sees `execute_trade` in the next `tools/list`

Workflow enforcement at the protocol level. No prompt engineering needed.

### Dynamic Check

Run arbitrary logic:

```ts
import { check } from "@stratus/mcp-aws";

server.tool("update_deal", {
  gate: check(
    (ctx) => ctx.auth.claims.org === "acme",
    "Only Acme employees can update deals",
  ),
}, handler);

// Async checks work too
server.tool("check_quota", {
  gate: check(async (ctx) => {
    const usage = await getUsage(ctx.auth.subject);
    return usage < 1000;
  }, "Quota exceeded"),
}, handler);
```

### Rate Limiting

```ts
import { rateLimit } from "@stratus/mcp-aws";

server.tool("expensive_op", {
  gate: rateLimit({ max: 10, windowMs: 60_000 }), // 10 per minute
}, handler);
```

Counters are per session + tool combination. Window resets after `windowMs`.

## Composing Gates

### `all(...)` — Every gate must pass

```ts
import { all, role, requires, rateLimit } from "@stratus/mcp-aws";

server.tool("approve_discount", {
  gate: all(
    role("sales-manager"),
    requires("review_discount"),
    rateLimit({ max: 10, windowMs: 3_600_000 }),
  ),
}, handler);
```

Returns the first failure.

### `any(...)` — At least one gate must pass

```ts
import { any, role } from "@stratus/mcp-aws";

server.tool("view_report", {
  gate: any(role("admin"), role("analyst")),
}, handler);
```

## Gate + Tier Matrix

| Tier | Gate | Behavior |
|------|------|----------|
| `always` | none | Everyone sees it, everyone calls it |
| `always` | `role` | Everyone sees it, only permitted users call it |
| `discoverable` | none | Must search to find, then anyone calls it |
| `discoverable` | `role` | Must search to find, only permitted users call it |
| `hidden` | `requires` | Invisible until prerequisite met, then auto-promoted |
| `hidden` | `role` + `requires` | Invisible until prerequisite met, then only permitted users |

## Denial Response

When a gate blocks, the tool returns structured JSON the agent can act on:

```json
{
  "error": "Permission denied",
  "reason": "Requires one of roles: admin, sales-manager",
  "hint": "Ensure the user has the appropriate role assigned."
}
```

For prerequisite gates:

```json
{
  "error": "Permission denied",
  "reason": "Requires \"review_trade\" to be called first",
  "hint": "Call the \"review_trade\" tool before using \"execute_trade\"."
}
```
